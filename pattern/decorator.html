装饰者模式是指在不改变原有对象的基础之上，将功能附加到对象上，提供了比继承更有弹性的替代方案

显然，煎饼的做法不仅仅只有这么几种，例如我们的材料后续加了辣条，那么就有辣条煎饼，辣条鸡蛋煎饼和辣条火腿煎饼等等。我们需要对每种煎饼的做法都实现一个子类，那么类的数量也未免太多了。

利用继承设计子类的行为，是在编译时静态决定的，而且所有的子类会继承相同的行为。但是我们使用组合的做法扩展对象的行为，就可以在运行时动态的进行扩展。

所以，在这里我们需要采取组合的形式：我们以煎饼为主体，然后在运行时以其他材料去”装饰”煎饼。比如，如果现在我需要一个辣条鸡蛋煎饼，那么分为以下几个步骤。

  1. 我们需要一个煎饼
  2. 用辣条装饰这个煎饼
  3. 用鸡蛋装饰这个煎饼
  4. 调用获取价格方法获取到相应的价格

装饰模式UML

装饰者模式中有四个角色：

  • Component：定义的基类，被装饰的原始对象，用来定义装饰者和被装饰者的基本行为。
  • ConcreteComponent：被装饰的具体对象，也就是基类的基本实现类。
  • Decorator：抽象装饰者，装饰基类，其内部有指向基类的引用。
  • ConcreteDecorator：装饰者具体实现类，也就是对抽象装饰者的具体实现。

在我们的案例中，Battercake就是装饰模式中的基类，而BaseBattercake则是被装饰的具体对象，

BattercakeDecorator是装饰者的抽象类，SpicyStripsDecorator和EggDecorator等类就是具体的装饰类。

装饰模式在Java IO中的应用

Java 中的流按照流的流向来分，可以分为输入流和输出流；也就是InputStream和OutputStream。

我们这里看InputStream的实现。

如图所示，InputStream作为装饰模式的基类，有三个子类实现作为被装饰的具体对象。

  1. FileInputStream：建立一个与文件有关的输入流。接受一个File对象作为流的源。
  2. StringBufferInputStream：将一个字符串缓冲区抓换为一个输入流。接受一个String对象作为流的源。
  3. ByteArrayInputStream：字节数组输入流在内存中创建一个字节数组缓冲区，从输入流读取的数据保存在该字节数组缓冲区中。接受一个Byte数组作为流的源。

而FilterInputStream就是抽象装饰类，他的子类就是具体的装饰类。

  1. PushbackInputStream：可以将已读取的直接“推回”输入流中。
  2. BufferInputStream：用来从硬盘将数据读入到一个内存缓冲区中，并从此缓冲区提供数据。
  3. DateInputStream：提供基于多字节的读取方法，可以读取原始数据类型的数据。
  4. LineNumberInputStream：提供带有行计算功能的过滤输入流。

我们可以发现，和我们原来的煎饼设计相比，并没有很大的差异。既然我们已经知道了装饰模式，并且也看过了InputStream的类图，那么我们可以自己来创建一个装饰者，把输入流的内的所有大写字符都转为小写字符。

以InputStream为例，可能大家初次接触的时候无法轻易的理解他的很多子类，但是如果知道了这些类是用来包装InputStream的以后，就会变得简单很多了。

同样的，这也是装饰模式的缺点，使用装饰模式的话，常常会造成设计中会出现大量的小类；增加了系统的复杂度，并且对于多次装饰的对象，对于排查错误时需要逐级排查，较为繁琐。

当然装饰模式提供了一种比继承更加灵活的扩展方式，所以在设计时也是可以考虑使用的。
